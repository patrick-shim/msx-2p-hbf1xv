## DISPOSITION: **PARITY** (split boundary delta = 0 lines; gate was ≤ 2)

## 1. Scenario

The §2.6 synthetic split protocol, zero game assets on either side: GRAPHIC4, per-row-unique
signature VRAM (row r byte c = (r+c) & 0xFF over all 256 rows of page 0), IE0+IE1 enabled,
R#19 = 80 / R#23 = 0, an IM1 handler that dispatches on S#1 FH — line-interrupt path reads S#1
(clears FH) and writes R#23 = 128; VBlank path reads S#0 and re-arms R#23 = 0 AND R#19 = 80
(mandatory under the both-references `(R#19 − R#23) & 0xFF` match relation,
references/openmsx-21.0/src/video/VDP.cc:527-529; references/fmsx-60/source/fMSX/MSX.c:2094-2104).

- Side A driver: `hbf1xv_m32_split_screen_system_test.exe --dump-frame` (the system test's real
  arm; the no-argv ctest invocation is unchanged). Artifact: `build/m32-ab/ours_split.frame`,
  committed render `debug/frames/m32-split-ab-ours.png`.
- Side B driver: the same protocol as a 512-byte Z80 program (org 0xC000, generated by
  `tools/gen-m32-split-probe.py`) injected into a live BASIC session via the openMSX debugger
  Tcl surface (`debug write memory` + `reg pc` — the same mechanism `tools/openmsx-ab-smoke.ps1`
  established), with its own page-0-to-RAM slot dance, IM1 hook install, and Z80-side VRAM
  signature fill. Capture: `screenshot -raw` (320×240, active area auto-located at

## 2. Comparison gate (region-structural, palette/scaler-independent)

Byte-level color comparison is meaningless across the two pipelines (different palette
conversion/scaler). Both sides are instead analyzed independently through two structural
properties of the signature content, using pixel EQUALITY only (no color knowledge):

- **P1 shift chain**: within one scroll region, display row y+1 equals display row y shifted
  left by exactly 2 pixels (source rows are +1-byte shifts of each other). Chain BREAKS recover
  the region structure: a split frame has exactly ONE break (the split boundary); a no-split
  frame has ZERO.
- **P2 offset-128 wrap**: display row 128+k equals display row k pixel-for-pixel for k ∈ [0,64)
  iff the bottom region runs at vertical scroll 128 (mod-256 wrap).

Both sides place the split boundary at display line **82** = R#19 (80) + ~2 lines of IM1-accept
+ handler-OUT-chain latency — identical interrupt-delivery latency behavior on both sides at
line granularity. Side B live counters at capture: `fh=24 vb=24` (one line interrupt and one
VBlank per frame — the handler protocol running for real on the reference), end-state
`r0=22 (0x16 = G4|IE1) r19=80 r23=128`.

## 4. Adversarial arms (both PASS)

- **Comparator self-check** (M28/M29 precedent): a copy of side A with one region corrupted
  (rows 100-139, x 0-63 XOR 0x1F) is FLAGGED (42 chain breaks, wrap drops to 52/64). Exit 0.
- **IE1-off probe on the reference** (`m32_split_basic_noie1.bin`, identical bytes except
  R#0 = 0x06): openMSX shows `fh=0` (the handler NEVER sees FH), ZERO chain breaks, 0/64 wrap —
  NO split. This independently corroborates our own machine-level IE1-off no-op contract
  against real openMSX, in addition to the system test's own adversarial arm.

## 5. Bring-up findings (recorded honestly; both encoded into the shipped harness)

1. **R#8 VR bit**: the probe initially wrote R#8 = 0x02 (SPD only), clobbering the VR bit
   (bit 3, 64K-DRAM organization) that BASIC's own SCREEN 5 sets. openMSX models VR-dependent
   VRAM addressing, so its display fetched scrambled addresses (uniform screen) even though
   the `VRAM` debuggable read back our fill correctly. Fixed: R#8 = 0x0A (VR|SPD). Our own
   renderer does not model VR (stored-only bit) — no side-A impact; recorded as a reference
   nuance.
2. **`set throttle off` staleness**: openMSX frame-skips aggressively when unthrottled, and
   `screenshot -raw` then returns a STALE frame (we captured a pre-fill frame twice). The
   harness re-enables throttle for ~1 emulated second before every capture.

## 6. Precision disclosure (package §2.5, encoded in the ≤2-line gate)

Our poll fires at the first instruction boundary at-or-after the raster ENTERS the matched
line; openMSX raises FH at the matched line's right border (VDP.cc:913-923). At line
granularity the observed boundary delta was 0.
