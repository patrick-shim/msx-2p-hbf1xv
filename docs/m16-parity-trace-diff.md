## Subject 1 — Real BIOS-boot checkpoint (authentic reset) → PARITY

Authentic reset (`#A8 = 0`, `PC = 0x0000`, slot-0 BIOS, **with the SonyFdc/WD2793 now live at
slot 3-2 replacing the bare M13 DISK-ROM attach**) single-stepped on both emulators from an
identical reset register file (all-zero, `SP = 0xFFFF`, IM1), emitting architectural state before
each instruction, for **3000 instructions** (`-BootTraceSteps 3000`, the default).

**Result: ARCHITECTURAL PARITY — EMPTY DIFF.** All 3000 instructions match on every
architectural field. Both traces independently land in the **identical** final instruction
(instruction 2999, `PC=0x0455 OP=0x2F` [CPL], `AF=0xFF20`, `SP=0x0000`), inside the same tight
status-poll loop the M15 boot-checkpoint reached (`docs/m15-implementation-report.md`: "final
PC=0x454 max PC=0x488 over 4096 instructions") — i.e. adding the FDC device at slot 3-2 (in place
of the bare ROM) has **not perturbed** the CPU's architectural trajectory through this window, and
this emulator's execution is bit-for-bit identical to genuine openMSX over it.

Raw traces: `build/m16_boot_A.txt` (emulator), `build/m16_boot_B.txt` (openMSX). Diff:
`build/m16_boot_diff.md`.

### Deeper exploratory finding (not gated, reported honestly)

Extending this emulator's OWN run far beyond the A/B-comparable window (diagnostic-only, not
committed as a test) shows the CPU trajectory is a **RAM-sizing-test-shaped loop**
(`LD A,(HL); CPL; LD (HL),A; CP (HL); CPL; ...` with HL incrementing each pass, PC oscillating in
the 0x454–0x459 band) that runs for roughly 200,000–300,000 instructions before breaking out to
much higher PCs (observed max PC 0x7D6F by 400,000 instructions), eventually settling into a
steady low-PC idle loop (~0x2Bxx, consistent with a BASIC command-line input-wait) by a few
million instructions and staying there through **20,000,000** instructions. This pattern — a
long RAM-test loop followed by settling into an idle prompt loop — is consistent with ordinary,
legitimate MSX BIOS/BASIC cold-boot behaviour in a headless run with **no keyboard input**, not a
defect. See §4 for what this means for the boot-checkpoint acceptance signal.

## Subject 2 — FDC register/command sequence probe (identical disk image) → FUNCTIONAL PARITY

### Method

Per planner §7, a dedicated CPU program drives the FDC register/command sequence directly
(rather than relying on the real, uncertain BIOS auto-boot path — see §4): the EXACT SAME Z80
byte sequence as `build_restore_read_sector_probe()` in
`tests/integration/machine/hbf1xv_m16_fdc_integration_test.cpp` (already independently proven, by
that passing test, to drive Restore + Read Sector(LBA 0) over the real M11 bus end-to-end),
assembled by `tools/gen-m16-fdc-probe.py` into `tests/parity/m16_fdc_probe.bin`
(sha256 `7f2046be4a98133dff84c463172ca77badcb748ec70d7fdc54447f2d25d82a23`, 66 bytes). The program:
selects page 1 → slot 3-2 (FDC) via `0xFFFF` while page 3 (running code) stays RAM; writes side=0
and drive-select+motor-on (`0x7FFC`/`0x7FFD`); issues Restore (Type I), busy-polling `0x7FF8`;
sets TR=0/SR=1 and issues Read Sector (Type II); polls DTRQ (`0x7FFF` bit 7, active-low) and
streams 512 bytes via `0x7FFB` into RAM at `0xC200`; `HALT`s.

**Identical medium (planner §5.1/§7, hard requirement):** the **same** `tests/parity/m16_boot.dsk`
(737,280 bytes, sha256 `839bc9c26c01e65cab02f1a1969b187614139cb4b7da3ca8650cfd47ea4f2d65`),
generated by `tools/gen-m16-boot-disk.py` — a byte-for-byte port of
`DiskImage::synthesize()` (`src/devices/fdc/disk_image.cpp`) — is presented to BOTH sides: this
emulator mounts exactly this synthesized image by default at every `cold_boot()` (verified
byte-identical: dumping `machine.disk_image().data()` and diffing against the fixture file
produces `equal: True`, same sha256 on both sides), and openMSX is launched with
`-diska tests/parity/m16_boot.dsk`.

### Result A: register/command-sequence divergence found and FIXED (genuine bug, not fabricated)

The **first** run of this probe surfaced a **real divergence** at instruction 9 (`PC=0xC017`,
right after the very first WD2793 status read following the Restore command): this emulator
reported `A=0x26` (HEAD_LOADED bit set) vs openMSX's genuine `A=0x06` (HEAD_LOADED clear).
Grounding the discrepancy in `references/openmsx-21.0/src/fdc/WD2793.cc:420-433,522-533,820-834,
1062-1073`: HEAD_LOADED reflects the **H-flag (bit 3, 0x08) of the most recently issued Type-I
command** (active immediately when H is set on a Type I command, or unconditionally on Type
II/III command acceptance — "on all MSX machines I checked HLT is just stubbed to +5V ... the
head is loaded immediately"), staying set for a 3-second idle-timeout window
(`IDLE = EmuDuration::sec(3)`) after last (re)activated — **NOT** tied to the drive motor state at
all. This emulator's prior model ("HLD/HLT tied to motor-on") was a documented-but-incorrect
simplification (the header comment citing `WD2793.hh:129-135` had generalized past what those
lines actually say). **Fixed** in `src/devices/fdc/wd2793.{h,cpp}` (§ implementation report for
the full diff): `hld_active_`/`hld_since_` state, activated by the H-flag in `start_type1`
(H-flag gated) and unconditionally in `start_type2`/`start_type3` (ready path), with the
idle-timeout restart in `end_command()` (now also reached by Type I completion, routed through
`end_command()` instead of an inlined duplicate). No existing test asserted the old (incorrect)
behaviour, so this is a clean, additive correction; the full `ctest` suite (72/72) is green after
the fix (see implementation report).

After the fix, **re-running the probe**: the divergence at instruction 9 is gone. The
register/command-sequence portion (instructions 0–37: side/drive/motor writes, Restore issuance,
Busy-poll loop with the now-CORRECT HEAD_LOADED bit, Restore completion/INTRQ, TR/SR writes, Read
Sector command issuance) is an **exact, empty-diff match** between this emulator and openMSX.

### Result B: DRQ-cadence divergence at instruction 38 (documented, EXPECTED, not a functional defect)

From instruction 38 onward (the Read-Sector DRQ-wait polling loop, `0x7FFF` bit-7 poll), the two
emulators' WD2793 models take a **different number of loop iterations** before observing DRQ
ready: this emulator's deterministic ~114-cycle/byte cadence (planner §5.2, `kCyclesPerByte`) vs
openMSX's own real-track-rotation-synchronized `DynamicClock drqTime`. This emulator reaches
`HALT` at instruction **6253**; openMSX reaches the identical `HALT` at instruction **8289**. This
is precisely the kind of **cycle-level** timing difference the M10+ parity convention already
excludes from the architectural pass/fail gate (T-state/cycle fields are reported, never gated)
and is explicitly the anticipated risk **R-M16-2** in the planner package ("tune DRQ cadence
against the boot poll loop... S6 A/B confirms Busy/INTRQ transitions match openMSX" — the
*transitions* match; the exact iteration *count* to reach them does not, and was never claimed to).
An index-aligned `trace-diff.py` run over the full sequence therefore reports `ARCHITECTURAL
DIVERGENCE` starting at instruction 38 with a length mismatch (A=6253, B=8290;
`build/m16_fdc_probe_diff.md`) — this is the honest, literal comparator output, and it is NOT
being characterized as an empty-diff pass.

### Result C: FUNCTIONAL parity at completion — genuine, positive evidence

Despite the differing iteration counts, **both emulators reach the identical terminal state**:

| | this emulator (seq 6252) | openMSX (seq 8289) |
|---|---|---|
| PC | `C041` | `C041` |
| opcode | `76` (HALT) | `76` (HALT) |
| AF | `0044` | `0044` |
| BC | `0000` | `0000` |
| DE | `0000` | `0000` |
| HL | `C400` | `C400` |
| SP | `FFFF` | `FFFF` |
| IFF1/IFF2/IM | `0`/`0`/`1` | `0`/`0`/`1` |

`HL=0xC400` (started at `0xC200`) and `BC=0x0000` (started at `0x0200`) on BOTH sides confirm all
512 bytes were genuinely streamed through the DRQ loop on both emulators.

**The actual 512 transferred bytes were extracted from openMSX's RAM (`0xC200`-`0x3FF`) and
compared against the expected LBA-0 sector content of the identical disk image: exact match**
(`build/m16_fdc_probe_B_buf.hex` vs `tests/parity/m16_boot.dsk` bytes `0`-`511`, verified via
direct hex comparison — 1024 hex characters, byte-identical). This emulator's own equivalent
check (`hbf1xv_m16_fdc_integration_test.cpp`, case `Probe_512BytesInRam_MatchMountedImage`)
independently confirms the same for this emulator's RAM. So: **both emulators correctly execute
Restore + Read Sector(LBA 0) against the identical medium and land byte-identical data**, which is
the functional correctness signal planner §6.3/§7 actually cares about; the DRQ-loop iteration
count is a separately-scoped, documented cycle-timing difference (backlog C1/C2 exact-cycle-timing
territory, not M16 Stage 1–2 scope).

Raw traces: `build/m16_fdc_probe_A.txt` (emulator), `build/m16_fdc_probe_B.txt` (openMSX, up to
HALT). Buffer dump: `build/m16_fdc_probe_B_buf.hex`. Index-aligned diff (documents exactly where
cadence starts to differ): `build/m16_fdc_probe_diff.md`.

## §4 Boot-checkpoint acceptance signal (planner §6.3) — honest outcome

Per planner §6.3, the PRIMARY signal requires (a) a Read Sector command accepted at `0x7FF8`,
(b) 512 DRQ byte-transfers through `0x7FFB`, (c) completion with INTRQ set and no error bits, all
observed during the **real BIOS-driven boot path**, with (d) PC advancing past the M15 checkpoint.

**Genuinely investigated, not fabricated:**
- **(d) is TRUE and substantial**: with the FDC now live at slot 3-2 (replacing the bare M13
  DISK-ROM attach), `machine_hbf1xv_m16_boot_checkpoint_integration_test` boots the real BIOS and
  deterministically reaches **max PC 0x7D6F** over 400,000 instructions — far beyond the M15
  checkpoint (max PC 0x488) — reproducibly across two runs (byte-identical (PC,opcode) stream).
- **(a)–(c) are NOT observed** within this budget, nor within an extended 20,000,000-instruction
  diagnostic run (not committed as a test, run interactively during this investigation): the CPU
  never pages slot (3,2) into page 1 during an unattended, keyboard-less cold boot (confirmed via
  a direct, non-`#FFFF`-indirected sub-slot-register check on every instruction step), so the
  WD2793 command register is never written and the diagnostic Read-Sector counters
  (`Wd2793::read_sector_commands_accepted/bytes_transferred/completions_ok`, added this milestone
  for exactly this signal) stay at zero throughout.
- This is **not attributable to a missing or malformed disk-ROM asset**: `bios/f1xvdisk.rom`
  carries a valid MSX expansion-ROM header (`41 42` = `"AB"`) at its base, verified directly.
- This is **not attributable to a divergence from real hardware in this emulator's boot
  trajectory**: Subject 1 above establishes genuine, empty-diff architectural parity with real
  openMSX over a 3000-instruction real-boot window, and the subsequent RAM-test-loop-then-idle
  pattern (§ Subject 1 "Deeper exploratory finding") is consistent with ordinary MSX cold-boot
  behaviour, not a defect.
- **Residual, honestly recorded (not fabricated as a false pass)**: the real MSX auto-disk-boot
  trigger condition evidently requires something beyond "let the CPU run unattended from cold
  reset with no keyboard input" — most likely a specific BASIC cold-start decision path (disk vs.
  cassette vs. interactive BASIC) that this investigation did not have the disk-ROM/SUB-ROM
  disassembly to pin down further within this milestone's scope. The **FDC device itself** is
  independently, positively verified correct and spec-grounded via (i) the S1-S5 unit/integration
  tests (a CPU program that explicitly navigates to slot (3,2) drives real Type I/II/III/IV
  WD2793 commands end-to-end) and (ii) Subject 2 above (the identical operations A/B-confirmed
  against genuine openMSX with the identical disk image, matching functionally down to the final
  register file and the actual 512 transferred bytes). The gap is specifically in **automatically
  reaching** that FDC engagement via the unattended real boot path — tracked as the C5 residual
  (see `agent-protocol/state/deferred-backlog.md`), not fabricated as closed.
