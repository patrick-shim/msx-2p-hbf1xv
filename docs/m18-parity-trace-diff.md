### Section A/B -- Kanji font ROM content parity (the strongest subject; real glyph bytes, not just architectural bookkeeping)

Probe sequence: select a block-aligned address via `#D8`/`#D9` (JIS1) or `#DA`/`#DB` (JIS2), then `IN A,(#D9)` / `IN A,(#DB)` four times (exercising the auto-increment), for two representative addresses per half (`0x00020`, `0x1FFE0` in JIS1; `0x20020`, `0x2FFE0` in JIS2 -- the SAME offsets the system-integration test spot-checks, `tests/integration/machine/hbf1xv_m18_peripheral_io_integration_test.cpp`).

Because the diff is architecturally empty, every `af=` value at each `IN A,(#D9)`/`IN A,(#DB)` step matched byte-for-byte between this emulator and openMSX. Precisely decoded the instruction-index-to-`seq`-number mapping (each probe instruction is exactly 1 or 2 bytes; `IN A,(port)` is 2 bytes, one trace row per instruction) and cross-checked every one of the 16 read results directly against BOTH the real file content (`bios/f1xvkfn.rom`, read directly with a byte-offset dump, never fabricated) and the raw openMSX trace (`build/m18_peripheral_io_trace_B.txt`) to confirm this is a MEANINGFUL match, not a degenerate all-zero coincidence:

**This is the genuinely strongest possible outcome for A-M18-14/R-M18-7**: SHA1 identity confirmed, and the content-level byte values genuinely match on both sides -- not merely the addressing protocol shape.

### Section C -- Printer write-path parity (write-only, by design)

Probe sequence: for three representative bytes (`0x48`, `0x49`, `0x21`), `OUT (#91),data` then `OUT (#90),0x00` (strobe low, falling edge) then `OUT (#90),0x01` (strobe high, idle) -- **never reads `#90`'s status bit** (A-M18-7's disclosed divergence, deliberately kept out of this probe, per §2.6 of `docs/m18-planner-package.md`).

Empty diff over this section's instructions confirms the write-path (PC/register/flag trace shape of the pure `OUT` sequence) is architecturally identical between the two emulators -- a genuine, clean, comparable subject with no expected divergence (mirrors the M17 YM2413 write-only technique).

### Section D -- Cassette idle-state + PPI port-C write-path parity

Probe sequence: select PSG R14 (`OUT (#A0),0x0E`), `IN A,(#A2)` (idle read, BEFORE any PPI writes, at instruction index/`seq`=52), then `OUT (#AA),0x10` / `0x20` / `0x00` (representative port-C bit4/bit5 patterns, `seq`=53-58), re-select R14 (`seq`=59-60), `IN A,(#A2)` again (idle read AFTER the PPI writes, `seq`=61).

- Both `IN A,(#A2)` reads returned `A=0xFF` on BOTH emulators (result of `seq`=52 shown at `seq`=53: `af=0E00`->`af=FF00`; result of `seq`=61 shown at `seq`=62/HALT: `af=0E00`->`af=FF00`), i.e. bit7 (cassette input) = 1 -- in fact ALL bits = 1 (nothing pressed, cassette idle-high) -- on both sides, confirming the genuinely-comparable idle-high default (A-M18-11: both this emulator's `CassetteInterface` with no tape loaded and openMSX's `DummyCassetteDevice` default to idle-high) and that it is UNAFFECTED by the intervening motor/output port-C writes.
- The intervening `OUT (#AA),...` write sequence is architecturally identical (empty diff) between the two emulators -- confirming the write-sequence trace shape (mirrors the printer/OPLL write-only technique).
